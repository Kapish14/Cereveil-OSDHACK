# Bootstrap Guardian domain state after Clerk authentication

Cereveil will create Guardian domain state in Convex after Clerk authenticates the Guardian rather than treating Clerk as the owner of domain objects. Guardian Mode will call an idempotent Convex bootstrap mutation after Clerk signup or sign-in instead of relying on Clerk webhooks for initial provisioning; that mutation resolves the Clerk identity using Convex's stable `identity.tokenIdentifier`, finds or creates only the Guardian Account, Household, and Guardian Device records, and returns the Guardian bootstrap state. Guardian Device records are keyed by a locally persisted app-generated installation ID within the Guardian Account, not by Clerk session ID or FCM token; bootstrap permits an existing installation but refuses a new installation once the Guardian Account already has two active Guardian Devices. Child Profiles, default policies, Enrollment Codes, FCM tokens, and supervision data are created by later explicit onboarding or registration flows. FCM token registration remains a separate operation because push tokens may be unavailable at signup and can rotate independently. This keeps authentication identity separate from supervision domain state while avoiding webhook signing, retry, duplicate-delivery, and local-development complexity in the first implementation.
